# Pre-baked layouts

Static assets served as `/baked-{label}.bin` by the demo. Generated by
the demo's "Bake current layout" button.

## How to bake (recommended: real Chrome, not headless)

```bash
npm run demo:dev
# open http://localhost:4174 in Chrome
# pick node count from sidebar (10k–1M)
# wait for the graph to settle visually (~3–5 s)
# set label (e.g. "1m-points"), tick "points only" if going large
# click "Bake current layout"
# wait for "saved · baked-{label}.bin · X.X MB"
```

The `.bin` file lands here in `demo/public/`.

## How to view a baked layout

In the same demo: set the label, click "Load baked layout". The hero
canvas tears down its live (sim-active) graph and rebuilds with the
baked positions, `enableSimulation: false`. This is what visitors
should see on the published demo — a settled graph at scale, no
spinning, instant pan/zoom.

## Why a baker that runs in the browser instead of headless

The baker uses the same `Graph.readbackPointPositions()` API that the
engine provides; running it in the browser means the simulation
settles naturally via the RAF loop (one step per frame, intervening
renders sync the storage buffer). Driving `graph.step()` in a tight
loop without rendering can over-shoot and collapse positions to
numerical zero — tested and ruled out.

In practice this means: keep the demo tab focused while baking. Don't
switch tabs (Chrome throttles background-tab RAF).

## Binary format

See `demo/src/baked-format.ts`. 32-byte header + Float32 LE positions
+ optional Float32 LE links.

## What ships in git

Only named "headline" bakes (`baked-1m.bin`, `baked-500k.bin`) are
checked in, gated by `demo/.gitignore`. Ad-hoc bakes stay local.
